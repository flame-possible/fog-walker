# Fog Walker — Local Data Pack World Expansion Design

> 작성일: 2026-06-28
> 상태: 승인된 방향 정리, 구현 계획 대기

## 1. 목표

Fog Walker는 현재 서울 행정동 기반 MVP로 완성한다. 동시에 데이터 모델과
컬렉션 구조는 이후 전세계 확장을 받을 수 있도록 준비한다.

전세계 확장은 서버 실시간 조회가 아니라 **선택 다운로드형 로컬 데이터팩**을
기본 전략으로 삼는다. 앱 설치 파일에는 시작 지역만 포함하고, 사용자가 다른
국가나 도시를 열 때 필요한 경계 데이터팩을 내려받아 로컬에 저장한다.

## 2. 제품 계층

사용자에게 보이는 지역 계층은 다음 5단계로 고정한다.

```
국가 → 지역 → 도시 → 구 → 동
```

- 국가: 대한민국, 일본, 미국 등.
- 지역: 행정 1단계. 한국의 특별시/광역시/도, 미국의 주, 일본의 도도부현.
- 도시: 행정 도시 또는 앱에서 탐험 단위로 쓰는 도시.
- 구: 도시 안의 구/자치구/ward/district.
- 동: 동/읍/면/neighborhood/locality 등 가장 작은 수집 단위.

국가별 행정 체계가 다르므로 내부 모델은 `level`, `parentId`, `kind`를 가진
트리 구조로 둔다. 화면에서는 위 5단계 명칭으로 정규화해서 보여준다.

## 3. 단계별 범위

### 3.1 현재 구현 범위

이번 구현은 서울 MVP 완성을 우선한다.

- 산책 거리 기록이 새 셀이 생길 때만 누적되는 문제 수정.
- My Info의 placeholder 값 정리: 국가 수, 프로필 사진, 즐겨 찾는 지역 표시.
- 프레임/문서 매핑 불일치 정리.
- 서울 데이터와 현재 화면은 계속 로컬 asset 기반으로 유지.
- 데이터 타입에는 향후 계층 확장에 필요한 필드를 보수적으로 추가한다.

### 3.2 향후 확장 범위

전세계 데이터팩 기능은 별도 단계에서 구현한다.

- 국가별 또는 도시별 데이터팩 다운로드.
- 데이터팩 목록/버전/용량/라이선스 메타데이터.
- 다운로드한 데이터팩의 로컬 저장, 삭제, 업데이트.
- 오프라인 사용.
- 국가별 품질 등급 표시.

## 4. 데이터팩 전략

앱 기본 번들은 작게 유지한다.

- 기본 포함: 서울 MVP 데이터.
- 선택 다운로드: 국가/지역/도시 단위 데이터팩.
- 캐시 정책: 사용자가 방문하거나 열어본 데이터팩만 로컬 보관.
- 삭제 정책: 앱 설정 또는 데이터 관리 화면에서 데이터팩 삭제 가능.

전세계 최하위 행정구역을 모두 앱 설치 파일에 넣는 방식은 피한다. 데이터가
수백 MB 이상으로 커질 가능성이 있고, 국가별 업데이트/라이선스 관리도 어렵다.

## 5. 데이터 소스 원칙

데이터 소스는 국가별로 다르게 조합할 수 있지만, 다음 원칙을 따른다.

- 상업 서비스와 재배포 조건이 명확한 데이터만 사용한다.
- 데이터팩마다 출처, 라이선스, 생성일, 버전을 저장한다.
- 가능하면 행정구역 데이터는 행정 계층을 우선한다.
- 생활권/관광권 컬렉션은 행정 데이터 위에 얹는 별도 컬렉션으로 다룬다.

후보 소스:

- geoBoundaries: 전세계 행정 경계 기본 후보.
- Overture Maps Divisions: 지역/도시/동네 계층 보강 후보.
- OpenStreetMap: 세부 동네 보강 후보. ODbL 의무를 별도로 검토한다.
- GADM: 라이선스 제약이 있어 기본 상업 서비스 소스로는 사용하지 않는다.

## 6. 앱 내부 모델 방향

현재 `RegionMeta`는 서울의 동 단위 메타데이터에 가깝다. 이를 한 번에 큰
리팩터링하지 않고, 서울 MVP에 필요한 호환성을 유지하면서 확장 가능한 형태로
옮긴다.

권장 필드:

- `id`: 전역 고유 id.
- `parentId`: 상위 지역 id.
- `countryId`: 국가 id.
- `level`: country, region, city, district, neighborhood.
- `kind`: province, state, metroCity, city, gu, dong, eup, myeon 등 원자료 타입.
- `nameKo`, `nameEn`, `localName`.
- `shape`, `bbox`.
- `dataPackId`: 어떤 데이터팩에서 온 지역인지.

사용자 진행 상태는 계속 별도 Hive 모델에 둔다.

- 방문 셀.
- 산책 기록.
- 지역별 해금/방문 횟수.
- 프로필.

클리어율, 면적, 업적 진행도는 저장하지 않고 파생한다.

## 7. 구현 우선순위

1. 현재 서울 MVP 품질 수정.
2. `RegionMeta`와 `RegionRepository`에 전세계 계층을 받을 수 있는 필드 추가.
3. 기존 서울 GeoJSON 생성 스크립트가 새 필드를 채우도록 수정.
4. Passport/City/Detail 화면은 기존 사용자 경험을 유지하되, 하드코딩 값을 줄인다.
5. 테스트로 기존 서울 동작이 깨지지 않았음을 확인한다.

## 8. 성공 기준

- 서울 MVP의 모든 기존 테스트가 통과한다.
- 산책 중 이미 방문한 셀 위를 걸어도 거리 기록은 계속 누적된다.
- 앱은 기존 서울 asset만으로 계속 동작한다.
- 서울 지역 데이터가 `국가 → 지역 → 도시 → 구 → 동` 계층으로 해석 가능하다.
- 이후 데이터팩 다운로드 기능을 추가할 때 기존 사용자 진행 데이터 구조를 크게
  깨지 않아도 된다.

## 9. 이번 단계에서 하지 않는 것

- 실제 전세계 데이터 다운로드 기능.
- 서버/API 구현.
- 데이터팩 스토어 UI.
- 국가별 전체 데이터 수집.
- OSM/geoBoundaries/Overture 자동 파이프라인.

이번 단계는 서울 MVP 완성과 확장 가능한 로컬 데이터팩 모델의 기반 마련까지만
다룬다.
